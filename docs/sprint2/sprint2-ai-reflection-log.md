# Sprint 2 AI 反思日志

更新时间：2026-07-14

记录格式：每条按 **问了什么 → AI 给了什么 → 我们改了什么 → 验证证据** 四段组织。验证证据用于说明人工判断已经通过测试或真机反馈闭环。

## 安卓书写优化

- **问了什么**：为什么不跟笔，画圆为什么有甩尾和折角？
- **AI 给了什么**：输入采样、预测点、平滑曲线和渲染路径分析。
- **我们改了什么**：分 P0/P1 实施；旧路径效果不佳后隔离新输入模型，并保留可调参数。
- **验证证据**：压感、轮廓、输入模型单测；安卓真机反馈。

## 实时协作笔迹

- **问了什么**：如何从“抬笔后同步”改为笔画过程同步？
- **AI 给了什么**：增量元素广播、发送合并和最终帧保证方案。
- **我们改了什么**：未采用全场景高频上传；改为发送实时笔迹帧，阻塞时只保留最新中间帧，最终帧必须送达。
- **验证证据**：`collaboration_repository_sync_test.dart`。

## 协作热路径优化

- **问了什么**：大文件双方同时编辑为什么卡顿和丢同步？
- **AI 给了什么**：拆分本地保存、封面和协作调度，引入 ChangeAccumulator 与接收合并窗口。
- **我们改了什么**：排除未经验证的“二进制膨胀 6 倍”等假设，改为先采集指标、再分阶段实施。
- **验证证据**：accumulator 单测；200 元素增量广播测试。

## PDF 导入持久化

- **问了什么**：多页 PDF 第二次打开时图片为什么消失？
- **AI 给了什么**：场景内容、图片引用和平台存储差异分析。
- **我们改了什么**：未大改数据库；修正大场景内容存取路径，并保留 Android/鸿蒙适配边界。
- **验证证据**：PDF 导入与场景仓库大内容回归测试。

## 系统分享与文件打开

- **问了什么**：`.markdraw/.excalidraw` 如何分享并让系统“打开方式”识别？
- **AI 给了什么**：Android Intent、鸿蒙 Want/Skills、入站队列和文件名传递设计。
- **我们改了什么**：分平台适配，公共层只接收标准载荷；补充后台唤醒和文件标题传递。
- **验证证据**：分享载荷、Android/鸿蒙服务与外部文档入站测试。

## 鸿蒙协作兼容

- **问了什么**：创建房间和账号区域为什么报 `unsupported_platform`？
- **AI 给了什么**：定位安全存储、HTTP 和平台插件缺失路径。
- **我们改了什么**：将平台能力收敛到适配层，公共协作状态机保持一致。
- **验证证据**：鸿蒙真机创建/加入房间；协作配置测试。

## 协作优化方案选型

- **问了什么**：三份 AI 研究文档（claude/codex/zcode-best）哪份最好？
- **AI 给了什么**：逐项对比 LWW/CRDT、ACK 机制、CSR、压缩等关键技术点。
- **我们改了什么**：选定 codex-best 为主线，吸纳 zcode 性能项与 claude 运行环境补强。
- **验证证据**：`socket_io_client 3.1.6` 源码验证 CSR 已支持 pid/offset；`sendBytes` 二进制传输。

## Phase 0 热路径修复

- **问了什么**：计划能否实现 codex-best 第一阶段？关键隐患在哪？
- **AI 给了什么**：审计出 5 个阻塞问题：syncAll→sceneInit、undo/redo 未写回、签名未传播、version 表缺 nonce、快照未等 in-flight。
- **我们改了什么**：逐项修复后由 Claude Code 执行，再审计确认 5 项全部闭合。
- **验证证据**：12 个提交、accumulator/reconciler/发送串行化单测全过。

## 安卓创建房间超时

- **问了什么**：安卓创建协作房间报 `TimeoutException after 0:00:15`，鸿蒙正常，为什么？
- **AI 给了什么**：排查 `encrypted_scene_store.dart` 的 15s 超时、`http.Client` 双端一致、`usesCleartextTraffic` 已设。
- **我们改了什么**：定位为网络层问题（端口 48931 在安卓端网络被封锁），非代码缺陷；建议改端口或测试浏览器可达性。
- **验证证据**：鸿蒙正常；代码无 Phase 0 改动；待安卓浏览器/蜂窝网络验证。

## 编辑器设置

- **问了什么**：哪些工具、手势、手写笔和文档设置最容易落地？
- **AI 给了什么**：复用控制器状态和 `LocalSettingsRepository` 的分批方案。
- **我们改了什么**：先实现已有开关和状态记忆，不引入新的状态管理库。
- **验证证据**：偏好 JSON、安全默认值与控制器应用测试。

## 工具栏导入图片

- **问了什么**：点「导入图片」为什么先弹「文件管理/相册」选择来源？
- **AI 给了什么**：定位到鸿蒙 `DocumentViewPicker` 会列出来源、安卓 `file_picker` 走 `ACTION_PICK`。
- **我们改了什么**：鸿蒙改用 `PhotoViewPicker`（`@kit.MediaLibraryKit`）、安卓改用 `Activity.startActivityForResult` + Photo Picker，双端直接进相册。
- **验证证据**：鸿蒙/安卓真机导入图片直进相册；`flutter analyze` 通过。

## 设置页未实现功能

- **问了什么**：设置页哪些分区是占位、实现优先级怎么排？
- **AI 给了什么**：按“成本×价值”分档，识别 `?section=` 路由参数未消费等隐性缺陷。
- **我们改了什么**：先落地第一档（其他/隐私/帮助反馈/路由修复），网盘备份先选型后再做。
- **验证证据**：设置页四个分区内容填充；`?section=` 路由参数生效；`flutter analyze` 通过。

## 网盘备份选型

- **问了什么**：百度网盘 vs 坚果云 vs 自建，哪条路最省事？
- **AI 给了什么**：对比 OAuth 准入门槛、回调复杂度、依赖和代码量。
- **我们改了什么**：否决百度网盘（OAuth 审核与移动端回调不确定）；选定坚果云 WebDAV（零依赖、复用 `http.Client`、复用加密）。
- **验证证据**：选型结论待验证——申请 AppKey/跑通 OAuth 前不写代码。

## 房主退出限制

## 反思结论

1. **AI 的核心价值是定位与拆解**。AI 最有价值的部分是快速定位跨平台调用链，并把问题拆成可验证假设。
2. **人工必须决定改动边界**。协作协议、数据库和跨端适配等关键路径，不能直接实现未经抓包、测试或真机验证的推断。
3. **书写与协作要联合回归**。本 Sprint 曾出现“改善局部书写体验但破坏协作”的回归，后续编辑器输入改动必须同时执行书写与协作回归测试。
4. **原生改动必须全量重编**。跨平台原生改动（鸿蒙 `.ets`、安卓 `Kotlin`）热重载不生效；AI 给出的 API 命名空间要查官方文档核对（如鸿蒙 `PhotoViewPicker` 属 `@kit.MediaLibraryKit` 而非 `@kit.CoreFileKit` 的 `picker`）。
5. **选型先验准入门槛**。功能选型要先验证“准入门槛”再写代码。网盘备份先否决百度网盘 OAuth 路径、确认坚果云 WebDAV 无准入门槛后，才进入实施，避免返工。
6. **方案对比要落到源码**。多 AI 方案对比不能只看文档措辞，要落到源码级证据。对比三份协作研究文档时，最终靠 `socket_io_client 3.1.6` 源码确认 CSR 的 pid/offset/recovered 均已实现，推翻了“Dart 不支持 CSR”的初始判断——grep 返回零匹配不代表能力不存在，可能是搜索方式有误。
7. **计划审计阻断阻塞**。Phase 0 计划在 5 轮审查中累计发现 5 个阻塞问题（syncAll→sceneInit 误传 bypass、undo/redo 未写回控制器、onSceneChanged 签名传播不全、version 表缺 versionNonce、快照未等 in-flight），全部在编码前修复，避免了“实现完再返工”的高成本。
8. **平台差异先排代码再查网络**。安卓创建房间 15s 超时而鸿蒙正常，最终定位为端口封锁而非 Flutter HTTP 客户端缺陷——`encrypted_scene_store.dart` 在 Phase 0 零改动是排除代码问题的关键证据。
