**四、遇到的问题与困难**

- **技术问题：**
  - **书写体验和协作同步存在天然冲突。** 高频笔迹输入既要本地低延迟渲染，又要避免协作广播过多导致卡顿。
  - **已尝试的解决方案：** 将本地书写、增量广播、队列合并和最终帧送达分开处理，并通过协作同步队列和增量应用优化响应速度。
  - **需要帮助或待解决：** 无，但后续所有书写模型改动都需要同步验证协作场景。
  - **Saber 内核迁移需要兼顾运行态体验和 Excalidraw 格式兼容。** 如果直接替换数据模型，会影响现有保存、协作和导入导出链路。
  - **已尝试的解决方案：** 采用 Saber 负责输入、笔迹、页面和渲染，Excalidraw JSON 继续作为保存、协作和导入导出的统一格式，并通过桥接层投影。
  - **需要帮助或待解决：** 后续扩展复杂元素时仍需严格维护桥接语义。
  - **全局识别和智能排版对模板几何高度敏感。** 横线纸、古籍版式、竖排页面和跨行笔迹会放大识别结果错位问题。
  - **已尝试的解决方案：** 按纸型锚点、会话和行块拆分笔迹，增加识别进度反馈，并针对模板行吸附、线间距和文字对齐反复修正。
  - **需要帮助或待解决：** 需要在更多真实笔记样本上继续观察识别和排版质量。
- **非技术问题：**
  - 结项阶段代码修复、验收文档、测试证据和 AI 使用记录集中推进，信息量较大。
  - **已采取的措施：** 按质量门禁、AI 反思、Sprint2 安全加固和个人周报分目录整理材料，尽量让提交记录、文档证据和实际功能保持一致。

**六、心得体会与学习收获**

- **本周学习到的新知识/技能：**
  - 深入理解了手写白板中输入采样、压感、路径缓存、画布渲染和协作同步之间的关系。
  - 学习了如何在不破坏协作格式的前提下迁移白板运行态内核，即通过桥接层同时保留 Saber 体验和 Excalidraw 数据兼容。
  - 熟悉了多模态 OCR、MyScript 识别、公式 LaTeX 渲染和智能排版在真实白板场景中的落地难点。
  - 掌握了 Sprint 质量门禁材料的组织方式，包括核心模块覆盖率、极端输入、性能数据和 AI 使用反思。
- **对项目或开发流程的思考：**
  - 白板类应用的核心复杂度来自数据模型、输入体验、渲染性能、协作协议和跨端适配的耦合，不能只把它当作普通画布 UI。
  - 大型重构必须先明确不变量。本项目中 Excalidraw JSON 是保存和协作的不变量，Saber 只替换运行态能力，因此风险可控。
  - AI 可以帮助快速定位问题和拆解方案，但关键边界仍要由开发者判断，尤其是许可证、平台能力、协作协议和验收证据。
- **自我改进方面：**
  - 后续做白板内核级改动时，应更早建立端到端验收清单，避免只验证单机书写而遗漏协作、导出和跨端。
  - 对智能识别和排版这类依赖外部服务的功能，需要保留更多真实样本和复现记录，方便持续评估效果。
  - 项目后期应继续保持文档和提交同步，让技术路线、风险取舍和验证证据都能被团队成员快速理解。
