**四、 遇到的问题与困难**

*   **技术问题：**
    *   **语音转文字各端适配不一致**：Android、鸿蒙和 Web 对系统语音识别能力、权限和返回结果的支持不同，不能直接复用同一实现。
    *   **鸿蒙端文件系统接口特殊**：文件选择、路径获取和存储插件与标准 Flutter 实现存在差异，部分通用插件无法直接使用。
    *   **协作同步优化会影响服务端部署**：调整广播频率、消息合并和快照策略后，需要同时关注 Flutter 客户端与 Go Socket.IO 服务端的协议兼容、超时和部署状态。
    *   **AI 模型接口兼容性差异**：不同 OpenAI 兼容模型对工具调用、思考内容和返回字段的实现不完全一致，导致思维导图生成或动作解析失败。
    *   **WebDAV 跨端限制**：移动端可直接使用 WebDAV 协议备份，但 Web 端会受浏览器 CORS 限制，无法直接调用部分 WebDAV 方法。

*   **非技术问题：**
    *   鸿蒙相关资料、插件示例相对较少，使用时常常会因为插件不适配报错。
    *   **已采取的措施：** 浏览官方文档、社区案例和真机日志交叉验证。

**六、 心得体会与学习收获**

*   **本周学习到的新知识/技能：**
    *   学习了 OpenAI 兼容接口的配置、连通性验证和工具调用适配方式，认识到不同模型的接口兼容程度不能仅依据文档判断，必须通过实际请求验证。
    *   进一步了解了 Flutter 在 Android、鸿蒙和 Web 端的差异，尤其是语音识别、文件系统、安全存储、网络请求和系统服务卡片等平台能力的适配方式。
    *   学习了 WebDAV 的目录创建、上传、下载和冲突处理流程，并认识到移动端网络请求可用不代表 Web 端一定可用，浏览器 CORS 策略会直接限制实现方案。
    *   在协作优化中理解了批处理、消息合并、节流和快照降频对实时白板性能的作用，也认识到客户端性能优化必须结合服务端部署和真实网络环境验证。

*   **对项目或开发流程的思考：**
    *   AI Agent 的价值不只是调用模型生成文本，更重要的是将模型输出限制为可校验、可预览、可应用的白板操作，避免不确定的模型结果直接影响用户数据。
    *   跨端功能应尽早在目标平台验证。先在 Android 上完成不代表鸿蒙和 Web 可以直接使用，平台差异如果拖到最后处理，往往会扩大修改范围。
    *   协作、语音、备份等功能都依赖外部环境，开发时应增加日志、连通性测试和明确的降级提示，减少用户只能看到“功能失败”却无法判断原因的情况。

*   **自我改进方面：**
    *   后续新增功能前先明确平台边界、依赖服务和验收方式，避免开发完成后才发现某个平台缺少必要能力。
    *   进一步加强测试记录和提交说明，将真机验证结果、复现条件和回归范围同步整理，提升多人协作时的问题定位效率。
