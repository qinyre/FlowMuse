# 协作实时同步问题诊断与根因分析

> 日期: 2026-07-12
> 分支: main
> 问题: 实时同步速度过慢，容易被动断开连接
> 分析范围: FlowMuse-App (Flutter) + FlowMuse-Server (Go) 全链路

---

## 1. 当前协作架构数据流

```
用户编辑 (EditorState)
  ↓ applyResult()
Scene 变更(不可变模型)
  ↓
CollaborationRepository.broadcastScene()
  ├─ getSyncableElements()     O(n) 过滤删除元素
  ├─ _changedElements()        O(n) 版本号对比
  ├─ CollaborationMessage.toBytes()  JSON 序列化
  ├─ _crypto.encrypt()         AES-GCM-128 加密
  └─ _transport.send()         Socket.IO 发送
       ↓
  [Socket.IO JSON: {encryptedBuffer: [byte,byte,...], iv: [byte,byte,...]}]
       ↓  (约 4-6x 数据膨胀)
  FlowMuse-Server (hub.go)
  forward() → client.To(room).Emit(client-broadcast, encryptedBuffer, iv)
       ↓  (零处理转发，不解密)
  其他客户端收到
       ↓
  _messageDecodeQueue (串行队列!)
  _crypto.decrypt() → CollaborationMessage.fromBytes()
       ↓
  reconcileRemoteScene() → applyRemoteScene()
       ↓
  EditorState.applyResult() → UI重绘
```

### 1.1 每20秒全量同步

```
_fullSceneSyncTimer (Timer.periodic, 20s)
  ↓
broadcastScene(syncAll: true)
  ↓ 发送全部元素 JSON，而非增量
```

### 1.2 快照持久化(HTTP旁路)

```
broadcastScene() → unawaited(_saveSceneSnapshot())
  ↓ HTTP PUT /api/rooms/{id}/scene
  │  body: {sceneVersion, sceneHash, baseSceneVersion, baseSceneHash, encryptedBuffer, iv}
  ↓ PostgreSQL UPSERT with optimistic lock (baseSceneVersion + baseSceneHash)
```

## 2. 根因分析(按影响程度排序)

### 🔴 P0: Socket.IO 二进制数据 JSON 编码膨胀(6x)

**位置**: [socket_io_realtime_transport.dart:339-348](socket_io_realtime_transport.dart)

```dart
// 当前实现: 将加密后的二进制数据作为 JSON 数组发送
socket.emit(
  volatile ? _eventServerVolatileBroadcast : _eventServerBroadcast,
  [
    roomId,
    {
      'encryptedBuffer': Uint8List.fromList(payload.encryptedBuffer),  // 每个 byte → "255," (3-4 字符)
      'iv': Uint8List.fromList(payload.iv),
    },
  ],
);
```

**问题**: Socket.IO 将 `Uint8List` 序列化为 JSON 数组 `[104,101,108,...]`，**每个字节膨胀为 2-4 个 ASCII 字符 + 逗号分隔符**。一个典型的场景更新消息(加密后约 2KB)在线上实际传输 **8-12KB**。

**服务端对应**: [hub.go:336-350](hub.go) 的 `asFrame()` / `asBytes()` 需要处理 JSON 数字数组的反序列化，进一步增加 CPU 开销。

**对比**: Socket.IO v4 原生支持 `Buffer` 类型传输，不需要 JSON 编码。服务端的 `parserTypes.BufferInterface` 已存在但客户端未使用。

**影响估算**: 消息体积膨胀 4-6x，对于手写笔高频编辑场景(每秒 30-60 笔划采样点)，带宽和 CPU 开销成倍增加。

---

### 🔴 P1: 消息处理串行队列阻塞

**位置**: [collaboration_repository.dart:414-422](collaboration_repository.dart)

```dart
_messageDecodeQueue = _messageDecodeQueue
    .then((_) => _handleEncryptedPayload(room, payload))  // 串行!
    .catchError(...);
```

**问题**: 每条收到的加密消息必须等前一条**完全解密 + 解析 + 回调分发**后才能开始处理。AES-GCM 解密是 CPU 密集型操作，串行化处理导致队列尾部消息延迟累加。

**场景**: 如果收到 3 条消息，每条解密 15ms，第三条消息的延迟 = 30ms(排队) + 15ms(解密) = 45ms。在 10 人协作房间中，队列深度可轻易达到 5-10 条。

---

### 🔴 P1: 每 20 秒全量场景同步

**位置**: [collaboration_repository.dart:703-712](collaboration_repository.dart)

```dart
void _startFullSceneSync() {
  _fullSceneSyncTimer = Timer.periodic(fullSceneSyncInterval, (_) {
    // 20秒一次，发送全部元素
    unawaited(broadcastScene(room: room, scene: _latestScene, syncAll: true));
  });
}
```

**问题**: 
- 场景包含 500+ 元素时，全量同步每次发送数百 KB 数据
- 即使场景完全没有变化，仍然触发加密 + 发送链路
- 与增量广播竞争带宽，可能阻塞正常的实时消息

**对比成熟方案**: Figma 全量同步仅发生在首次加入时，之后只发送 property-level delta。Google Docs 使用 revision log，只发送操作日志增量。

---

### 🟡 P2: 缺乏消息批处理/节流

**问题**: 每次 `broadcastScene()` 调用立即触发 `_send()`，没有合并窗口。

- 手写笔划: 一笔可能触发 10-30 次 element update(每次点追加),每次都独立发送
- 鼠标位置: 每次 pointer move 独立发送(虽然标记为 volatile,但仍消耗带宽)
- 拖拽移动: 每个 pointer 事件独立发送整个 element JSON

**对比成熟方案**: 
- Figma: 33ms 批处理窗口(client 端合并所有变更，30fps 发送)
- Yjs: 16ms coalescing 窗口
- Google Docs: ACK 规则自然合并(一次只能有一个 in-flight op)

---

### 🟡 P2: 连接稳定性不足

**位置**: [socket_io_realtime_transport.dart:95-98](socket_io_realtime_transport.dart)

```dart
final options = io.OptionBuilder()
    .setTransports(['websocket', 'polling'])
    .disableAutoConnect()
    .enableReconnection();  // 使用默认重连参数，未定制
```

**问题**:

| 参数 | 当前值(默认) | 建议值 | 说明 |
|------|-------------|--------|------|
| reconnectionDelay | 1s | 1s(start) | 初始延迟合理 |
| reconnectionDelayMax | 5s | 30s | 移动网络需要更长退避 |
| reconnectionAttempts | 无限 | 10 | 无限重试耗尽电量 |
| pingInterval | 25s | 20s | 略低于 NAT 超时 |
| pingTimeout | 20s | 10s | 更快速检测断连 |
| timeout | 20s | 10s | 初始连接超时 |

**额外缺失**:
- 无平台网络状态监听(Android `ConnectivityManager`, iOS `NWPathMonitor`)
- 无应用前后台切换时的连接管理
- 重连后无状态差量同步(直接重新 join，触发全量场景推送)

---

### 🟡 P3: O(n) 元素遍历开销

**位置**: [collaboration_repository.dart:686-701](collaboration_repository.dart)

```dart
List<Map<String, Object?>> _changedElements(elements) {
  return [
    for (final element in elements)  // O(n) 遍历
      if (!_broadcastedElementVersions.containsKey(_id(element)) ||
          _version(element) > _broadcastedElementVersions[_id(element)]!)
        element,
  ];
}
```

**问题**: 每次 `broadcastScene()` 都做 O(n) 全量扫描。配合 `getSyncableElements()` (也是 O(n)) 和 `getSceneVersion()` (也是 O(n) fold)，单次广播需要 3 次全量遍历。对于 1000+ 元素场景，每帧浪费数毫秒 CPU。

**优化方向**: 维护脏元素集合(dirty set)，变更时标记，广播时只遍历脏元素。

---

### 🟢 P3: 快照保存阻塞与冲突重试

**位置**: [encrypted_scene_store.dart:211-245](encrypted_scene_store.dart)

- 快照保存是 `unawaited` (不阻塞广播)，但内部 `_saveSceneResolvingConflict()` 在 409 冲突时做完整 round-trip: 加载全量 → 全量 reconciler → 重新加密 → 重新 PUT
- 每次快照保存都加密全量场景(不是增量)

---

### 🟢 P4: 服务端无优化转发

**位置**: [hub.go:147-182](hub.go)

```go
func (h *Hub) forward(client *socket.Socket, args []any, volatile bool) {
    // ... 验证 ...
    operator := client.To(socket.Room(roomID))
    if volatile {
        operator = operator.Volatile()
    }
    operator.Emit(EventClientBroadcast, frame.EncryptedBuffer, frame.IV)
}
```

**问题**: 纯转发，没有:
- 消息合并(短时间窗口内的多条消息合并为一条)
- 压缩(服务端不关心内容，但可以透明压缩 WebSocket 帧)
- 限流(单客户端发送过快时的保护)

## 4. 连接断开根因分析

### 4.1 Socket.IO 默认 pingTimeout 在移动网络的不足

Socket.IO v4 默认 `pingTimeout = 20000ms`。这意味着服务端发送 ping 后等 20 秒才判定断连。但移动网络 NAT 超时中位数约 128 秒。问题不是 pingTimeout 太短，而是:

1. **网络切换( Wi-Fi ↔ 4G/5G )**: IP 变化导致 TCP 连接直接断开，Socket.IO 需要检测到断连后才开始重连(依赖 ping 超时，最多 20s + pingInterval 25s = 45s 才检测到)
2. **应用后台**: 移动 OS 可能暂停 WebSocket 线程，恢复后连接已半开
3. **代理超时**: 如果使用了 Nginx/负载均衡器，其 proxy_read_timeout 可能低于 Socket.IO 的 pingInterval

### 4.2 重连后状态丢失

当前重连逻辑: 重连 → emit join-room → 服务端当做新用户 → 触发 `_sendSceneInitToNewUser()` → 全量场景推送。

**问题**: 重连期间错过的增量消息永久丢失，依赖 20s 全量同步来修复不一致。
