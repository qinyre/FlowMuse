# 成熟产品协作技术参考分析

> 日期: 2026-07-12
> 来源: 官方技术博客、学术论文、开源代码分析
> 目的: 为 FlowMuse 协作优化提供可借鉴的技术方案

---

## 1. Figma — 属性级 LWW + 中心化权威服务

**技术栈**: Rust 多人服务器 + DynamoDB WAL + 属性级 LWW + 小数索引

### 核心设计决策

| 决策 | 理由 |
|------|------|
| 拒绝 OT | 操作变换矩阵 O(n²) 复杂度，设计工具的元素操作远不止 insert/delete |
| 拒绝纯 CRDT | 向量时钟和墓碑在中心化架构中是浪费 |
| 属性级 LWW | 设计工具中用户极少同时编辑同一属性，真正冲突罕见 |
| 分数索引 | 避免 OT 的位置变换，base-95 编码高效 |
| 每文档一个 Rust 进程 | 从 TypeScript 重写后序列化快 10 倍 |

### 同步协议

```
连接 → 全量文档(仅一次) → 属性级增量(永久)
                                ↓
批处理窗口: 33ms (30fps)
                                ↓
服务端: 权威排序 + LWW 冲突裁决
                                ↓
广播到其他客户端
```

### 光标/存在感

- 同一 WebSocket 连接，独立 ephemeral 通道
- 33ms coalescing → 接收端 60fps 插值
- 硬上限: 200 光标/文件
- 不持久化(disconnect = 立即消失)

### 对 FlowMuse 的启示

1. **属性级 LWW 是白板场景的正确冲突粒度** — FlowMuse 当前是元素级 LWW，可以保持不变(白板比设计工具更粗糙，元素级已足够)
2. **批处理窗口**(33ms)是延迟和带宽的最优权衡
3. **存在感数据不持久化** — FlowMuse 的 mouseLocation/idleStatus 也是 volatile，这点正确
4. **分数索引** — FlowMuse 已经实现了自己的版本，但 SceneReconciler 使用字符串比较而不是数学插入，效率较低

## 2. Excalidraw + Yjs — CRDT 白板协作

**技术栈**: Yjs (YATA CRDT) + y-websocket/WebRTC + Awareness 协议

### 数据模型

```typescript
const ydoc = new Y.Doc();
const yElements = ydoc.getArray<Y.Map<any>>('elements'); // Y.Array + Y.Map per element
const yAssets = ydoc.getMap('assets');                     // Y.Map for assets
```

### 同步粒度

- 元素级同步(整个 element JSON 作为 Y.Map 中的一个 entry)
- 一个元素变更 → 整个 Y.Map entry 被标记为 dirty → 二进制 delta 编码
- 不是字段级同步(如只改 x 坐标，仍发送整个 element)

### Yjs 关键优化

| 优化 | 效果 |
|------|------|
| 行程编码(run-length) | 连续字符/操作合并为单个节点 |
| 二进制状态向量 | 用 Uint8Array 编码同步状态(非 JSON) |
| 16ms coalescing | 快速操作合并为单次 CRDT 事务 |
| Y.UndoManager | 每个用户独立 undo 栈 |

### Awareness 协议(光标同步)

- 独立的状态型 CRDT (CvRDT / LWW-Register)
- 每个 client 只有一个 entry，只能自己修改
- 逻辑时钟决定新旧
- 30 秒无心跳自动标记离线
- 状态变化时广播**全量** awareness 状态(非 delta)
- 优势: 简单，状态小(几十字节)

### 对 FlowMuse 的启示

1. **Yjs 的二进制 delta 编码效果极好** — FlowMuse 可以借鉴: 在加密前用更紧凑的格式编码变更
2. **Awareness 与文档数据分离** — FlowMuse 已经在做(volatile vs 非 volatile)
3. **Yjs 的元素级同步与 FlowMuse 一样** — 说明元素级粒度对白板合理
4. **Yjs + y-excalidraw 组合证明**CRDT 能很好兼容 Excalidraw 格式

## 3. Google Docs — OT Jupiter 算法 + Revision Log

**技术栈**: OT (Jupiter 变体) + 中心化 OT 引擎 + Event Sourcing

### 核心设计

#### ACK 规则(关键约束)
```
客户端一次最多只能有一个未确认操作在途。
在收到 ACK 之前，所有新编辑进 pending buffer。
收到 ACK 后，pending buffer 合并为一个操作发送。
```

**效果**: 自然实现了消息批处理 — 不需要显式 Throttle/Batching，ACK 规则本身就是批处理机制。

#### Revision Log 作为 SSOT
```
文档 = revision log 的物化视图
Revision 0 (空文档) + Op1 + Op2 + ... + OpN = 当前文档
```

**优势**:
- 完整修订历史(免费)
- 从任意时间点恢复
- 通过 Period Snapshot + Compaction 加速冷启动

#### 双引擎架构

| 引擎 | 位置 | 职责 |
|------|------|------|
| 服务端 OT | Google 服务器 | 对并发操作排序 + 变换 + 广播 |
| 客户端 OT | 浏览器 JS | 变换远程操作以兼容本地 pending 操作 |

**去掉任一侧都有 bug** — 这是 OT 的核心复杂性。

### 对 FlowMuse 的启示

1. **ACK 规则是优雅的批处理方案** — 但 FlowMuse 不适合直接套用(白板编辑不是字符流)
2. **Revision Log 作为 SSOT** — FlowMuse 的 `_saveSceneSnapshot` 已经实现了类似的概念(定期快照)
3. **OT 的复杂性不适合白板** — Google Docs 选择 OT 是因为文字编辑有严格的插入位置语义，白板元素的位置冲突远不如此敏感

## 4. Liveblocks — 托管协作后端 + Yjs 集成

**技术栈**: Cloudflare Edge Workers + WebSocket Rooms + CRDT Storage + Yjs Provider

### 架构

```
客户端 ←→ Liveblocks Room (Edge WebSocket 服务器)
                  ↓
            CRDT Storage (自动持久化)
```

### 核心概念

| 概念 | 说明 |
|------|------|
| Room | 有状态 WebSocket 服务器(部署在 CDN 边缘) |
| Storage | CRDT 数据存储(自动持久化) |
| Presence | 存在感数据(光标、选区) |
| Broadcast | 房间内广播事件 |
| Yjs Provider | 通过 `@liveblocks/yjs` 桥接 Yjs CRDT |

### 对 FlowMuse 的启示

1. **Presence vs Storage 分离** — FlowMuse 已经在做(volatile 标记)
2. **Yjs 桥接模式** — 如果未来引入 Yjs，参考 `@liveblocks/yjs` 的实现
3. **边缘部署** — FlowMuse 目前是单机部署，但 Go 后端迁移到多节点的复杂度低于 JS/Node.js

## 5. 跨产品对比总结

### 5.1 冲突解决策略选择

```
                  同步延迟要求
                  低 ←──────────→ 高
               ┌──────────────────────┐
          简单 │  LWW           LWW   │ 简单
          ↑    │  (Figma)    (FlowMuse)│    ↑
冲突解决  │    │  ─────────────────── │    │
复杂度    │    │  OT             CRDT  │    │
          ↓    │  (Google Docs) (Yjs)  │    ↓
          复杂 │                       │ 复杂
               └──────────────────────┘
                文字/代码      白板/设计
                
              FlowMuse 当前位于最优区间
              (白板场景 + 简单 LWW = 正确选择)
```

### 5.2 关键指标对比

| 指标 | Figma | Excalidraw/Yjs | Google Docs | Liveblocks | FlowMuse 当前 |
|------|-------|---------------|-------------|------------|--------------|
| 冲突算法 | LWW(属性级) | CRDT(YATA) | OT(Jupiter) | CRDT | LWW(元素级) |
| 全量同步 | 仅首次 | 仅首次 | 仅首次 | 仅首次 | **每 20s** ❌ |
| 消息编码 | 二进制 | 二进制(Uint8Array) | 二进制 | 二进制 | **JSON 数组** ❌ |
| 批处理 | 33ms | 16ms(事务) | ACK 规则 | Yjs 事务 | **无** ❌ |
| 存在感 | ephemeral | Awareness CRDT | ephemeral | Presence | volatile ✅ |
| 连接恢复 | 状态差量 | 状态向量差量 | revision log | Yjs 差量 | **全量场景** ❌ |
| 离线支持 | 有限 | 原生(Yjs) | Buffer+Reconcile | Yjs | **无** |

FlowMuse 的架构骨架(元素级 LWW + Excalidraw 兼容 + E2E 加密)是正确的。❌ 标记的是工程实现层面的问题，通过 Phase 1-3 优化可以修复。

## 7. 为什么不做 Yjs 集成

(现阶段)

1. **LWW 对白板已足够**: Figma 用 LWW 支撑了全球最大的设计协作平台，说明 LWW 不是瓶颈
2. **问题在工程实现不在算法**: 当前延迟主要来自 JSON 编码膨胀(6x)和串行解密，不是冲突解决不对
3. **Flutter 端 CRDT 生态薄弱**: Yjs 是 JS 生态，Flutter 需要用 `dart:js` interop 或 port — 两者都增加复杂度
4. **先穷尽简单优化**: Phase 1-3 的优化后性能很可能已经满足需求

**触发 Yjs 评估的条件**: Phase 1-3 完成后，5+ 人协作 P95 延迟仍 > 200ms，或离线编辑成为硬需求。
