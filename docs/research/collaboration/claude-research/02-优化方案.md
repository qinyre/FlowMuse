### 优化 1.1: Socket.IO 原生二进制传输 (P0)

**目标**: 消除 4-6x JSON 编码膨胀

**当前代码** ([socket_io_realtime_transport.dart:339-348](socket_io_realtime_transport.dart)):
```dart
// ❌ 当前: Uint8List 被 JSON 序列化为数字数组
socket.emit(
  volatile ? _eventServerVolatileBroadcast : _eventServerBroadcast,
  [
    roomId,
    {
      'encryptedBuffer': Uint8List.fromList(payload.encryptedBuffer),
      'iv': Uint8List.fromList(payload.iv),
    },
  ],
);
```

**优化方案**:

```dart
// ✅ 使用 Socket.IO 原生二进制传输
// Socket.IO 的 emit 支持直接传 Uint8List 作为二进制帧，
// 不走 JSON 序列化 -- 零膨胀
socket.emit(
  volatile ? _eventServerVolatileBroadcast : _eventServerBroadcast,
  roomId,                    // 第一个参数: roomId (字符串, JSON)
  payload.encryptedBuffer,   // 第二个参数: 加密数据 (二进制)
  payload.iv,                // 第三个参数: IV (二进制)
);
```

**服务端适配** ([hub.go:311-334](hub.go)):

当前服务端的 `parseBroadcastArgs` 已支持从二进制帧提取数据(`parserTypes.BufferInterface`)，只需调整参数顺序即可:

```go
func parseBroadcastArgs(args []any) (string, EncryptedFrame, bool) {
    if len(args) < 3 {
        return "", EncryptedFrame{}, false
    }
    roomID, ok := asString(args[0])       // roomId
    if !ok { return "", EncryptedFrame{}, false }
    encryptedBuffer, ok := asBytes(args[1]) // 二进制 encryptedBuffer
    if !ok { return "", EncryptedFrame{}, false }
    iv, ok := asBytes(args[2])             // 二进制 iv
    if !ok { return "", EncryptedFrame{}, false }
    return roomID, EncryptedFrame{EncryptedBuffer: encryptedBuffer, IV: iv}, true
}
```

**预期收益**: 消息体积减少 75-83%，编解码 CPU 减少 60%+。

### 优化 1.2: 并行解密 + 保序应用 (P1)

**目标**: 消除串行解密队列的延迟累加

**当前代码** ([collaboration_repository.dart:414-422](collaboration_repository.dart)):
```dart
// ❌ 串行: 每条消息等前一条解密完才开始
_messageDecodeQueue = _messageDecodeQueue
    .then((_) => _handleEncryptedPayload(room, payload))
```

**优化方案**:

```dart
// ✅ 并行解密，但按接收顺序应用
final _pendingDecrypts = <_DecryptTask>[];
bool _decryptLoopRunning = false;

void _enqueueDecrypt(CollaborationRoom room, EncryptedPayload payload) {
  final task = _DecryptTask(room: room, payload: payload);
  _pendingDecrypts.add(task);

  // 立即启动异步解密(不等待)
  task.decryptFuture = _crypto.decrypt(
    roomKey: room.roomKey,
    encryptedPayload: payload,
  ).then((bytes) {
    task.decryptedBytes = bytes;
  }).catchError((e) {
    task.error = e;
  });

  if (!_decryptLoopRunning) {
    _runDecryptLoop(room);
  }
}

Future<void> _runDecryptLoop(CollaborationRoom room) async {
  _decryptLoopRunning = true;
  while (_pendingDecrypts.isNotEmpty) {
    final task = _pendingDecrypts.first;
    try {
      // 等待这条消息解密完成
      await task.decryptFuture;
      if (task.error != null) throw task.error!;
      // 解密完成后立即启动下一条的解密(已经在跑了)
      // 这里只做顺序应用
      _applyDecryptedMessage(room, task.decryptedBytes!);
    } catch (e) {
      _addRepositoryError('协作消息解密失败：$e');
    }
    _pendingDecrypts.removeAt(0);
  }
  _decryptLoopRunning = false;
}
```

**关键点**:
- 解密是纯 CPU 操作，多线程(Isolate)并行安全
- 应用消息必须保序(依赖元素 version 的 LWW 语义)
- 使用 Future 提前启动下一条解密，应用侧仍按 FIFO

**预期收益**: 多用户场景下消息处理延迟降低 40-60%。

### 优化 1.3: 维护 Dirty Set 替代 O(n) 全量扫描 (P3)

**目标**: 消除每次广播的 3 次 O(n) 全量遍历

**当前代码** ([collaboration_repository.dart:686-695](collaboration_repository.dart)):
```dart
// ❌ 每次广播都全量遍历
List<Map<String, Object?>> _changedElements(elements) {
  return [for (final element in elements)
    if (!_broadcastedElementVersions.containsKey(_id(element)) || ...)
      element];
}
```

**优化方案**:

```dart
// ✅ 维护脏元素集合
final _dirtyElementIds = <String>{};

/// 在 applyResult 或 reconcileRemoteScene 之后调用
void _markDirty(String elementId) {
  _dirtyElementIds.add(elementId);
}

void _markDirtyAll(Iterable<String> elementIds) {
  _dirtyElementIds.addAll(elementIds);
}

List<Map<String, Object?>> _changedElements(List<Map<String, Object?>> elements) {
  if (_dirtyElementIds.isEmpty) return [];
  final changed = <Map<String, Object?>>[];
  for (final element in elements) {
    if (_dirtyElementIds.contains(_id(element))) {
      changed.add(element);
      _dirtyElementIds.remove(_id(element));
    }
  }
  return changed;
}
```

**联动**: `getSyncableElements()` 仍需保留(删除元素过滤)，但可将过滤范围限制在 `_dirtyElementIds` 内。

**预期收益**: 1000 元素场景下每帧 CPU 减少 2-5ms。

## Phase 2: 架构优化(3-5 天)

### 优化 2.1: 消息批处理/Coalescing (P2)

**目标**: 合并短时间窗口内的多次编辑为单次广播

**方案**:

```dart
/// 批处理配置
static const Duration _batchWindow = Duration(milliseconds: 50);
static const int _maxBatchSize = 50; // 窗口内最多累积 50 个变更

Timer? _batchTimer;
final Set<String> _batchDirtyIds = {};

Future<void> broadcastSceneBatched({
  required CollaborationRoom room,
  required ExcalidrawScene scene,
}) async {
  _latestScene = scene; // 总是更新最新场景引用
  
  // 标记脏元素
  for (final element in _changedElements(scene.elements)) {
    _batchDirtyIds.add(_id(element));
  }

  if (_batchTimer?.isActive ?? false) {
    // 窗口内已有定时器，本次变更已记录
    return;
  }

  _batchTimer = Timer(_batchWindow, () async {
    _batchTimer = null;
    final dirtyIds = Set<String>.of(_batchDirtyIds);
    _batchDirtyIds.clear();

    // 从最新场景提取脏元素的当前状态
    final elementsToSend = <Map<String, Object?>>[
      for (final element in _latestScene.elements)
        if (dirtyIds.contains(_id(element))) element,
    ];

    if (elementsToSend.isEmpty) return;

    await _send(
      room: room,
      message: CollaborationMessage.sceneUpdate(elements: elementsToSend),
    );
    _rememberBroadcasted(elementsToSend);
    _lastBroadcastedOrReceivedSceneVersion = _reconciler.getSceneVersion(
      _latestScene.elements,
    );
    _scheduleFileUpload(room);
    unawaited(_saveSceneSnapshot(room: room, scene: _latestScene));
  });
}
```

**对 volatile 消息的独立节流**:
```dart
// 鼠标位置: 最多每 50ms 发送一次
DateTime _lastMouseBroadcast = DateTime.now();
static const Duration _mouseThrottle = Duration(milliseconds: 50);

Future<void> broadcastMouseLocation(...) {
  final now = DateTime.now();
  if (now.difference(_lastMouseBroadcast) < _mouseThrottle) return;
  _lastMouseBroadcast = now;
  return _send(room: room, volatile: true, message: ...);
}
```

**预期收益**: 手写笔划场景消息量减少 70-90%，拖拽场景减少 80%。

### 优化 2.2: 取消定时全量同步 (P1)

**目标**: 消除每 20 秒的全量场景推送

**方案**: 将全量同步改为**状态向量差量校验**

```dart
/// 替换 _startFullSceneSync()
void _startConsistencyCheck() {
  _fullSceneSyncTimer = Timer.periodic(
    const Duration(seconds: 10), // 降频到 10s 且不再发全量
    (_) => _sendConsistencyCheck(),

}

Future<void> _sendConsistencyCheck() async {
  final room = _activeRoom;
  if (room == null) return;

  // 只发送场景版本哈希(极小,<100 bytes)

// 接收端: 收到 consistency check 后比对

      localElements: _latestScene.elements,

  }
}
```

**预期收益**: 消除周期性带宽尖峰，正常负载下几乎无 20s 全量同步开销。

### 优化 2.3: 新用户加入时的差量同步 (P2)

**替代当前 `_sendSceneInitToNewUser()`**:

当前方案: 新用户加入 → 发送全量场景

优化方案: 新用户加入 → 发送当前场景版本号 → 新用户从快照加载 → 只补发版本差量

```dart
Future<void> _sendSceneInitToNewUser(CollaborationRoom room) async {
  // 只发最新版本号，新用户自己从快照加载并请求差量
  await _send(
    room: room,
    message: CollaborationMessage.sceneVersionHint(
      sceneVersion: _reconciler.getSceneVersion(_latestScene.elements),
      sceneHash: _reconciler.hashElementsVersion(_latestScene.elements),
    ),
  );
}
```

新用户端流程:
1. 收到 `sceneVersionHint` → 从 `_sceneStore.loadScene()` 加载快照
2. 应用快照到本地场景(已经是最新的,大概率零差量)
3. 快照版本 >= hint 版本 → 无需额外同步
4. 快照版本 < hint 版本 → 等待下一条增量消息补全

### 优化 3.2: 快照增量保存 (P3)

**当前**: 每次 `_saveSceneSnapshot` 加密全量场景。

**优化**: 维护上次快照的元素状态，只对变更元素做增量合并。

```dart
Future<void> _saveSceneSnapshotIncremental({
  required CollaborationRoom room,
  required List<Map<String, Object?>> changedElements,
}) async {
  // 从服务端加载当前快照(可能是旧版本)
  final storedScene = await _sceneStore.loadScene(room);
  if (storedScene == null) {
    // 首次保存,全量
    await _sceneStore.saveScene(room: room, scene: _latestScene);
    return;
  }

  // 合并变更元素到存储场景
  final mergedElements = _reconciler.reconcile(
    localElements: changedElements,    // 只发变更
    remoteElements: storedScene.elements,
  );
  
  final mergedScene = storedScene.copyWith(
    elements: _reconciler.getSyncableElements(mergedElements),
  );

  await _sceneStore.saveScene(room: room, scene: mergedScene);
}
```

### 优化 3.3: 压缩传输 (可选)

如果 Phase 1 的二进制传输后还需要进一步压缩:

```dart
// 对加密后的数据进行 zlib 压缩再发送
import 'dart:io'; // gzip

List<int> compressPayload(List<int> encryptedBytes) {
  return gzip.encode(encryptedBytes);
}
```

**注意**: AES-GCM 密文是均匀随机的，压缩效果很差(几乎无压缩)。如果需要压缩，应在加密**之前**压缩明文 JSON。这是 AES-GCM + E2E 加密场景下的固有约束。

**替代方案**: 在加密前使用更紧凑的序列化格式(MessagePack 替代 JSON):

```dart
import 'package:msgpack/msgpack.dart';

List<int> toBytes() {
  // MessagePack 通常比 JSON 紧凑 20-40%
  return msgpack.encode(toJson());
}
```

### 优化 3.4: Yjs 集成评估 (长期可选)

**评估标准**:

| 因素 | 自研(当前方向) | 集成 Yjs |
|------|--------------|----------|
| 开发成本 | 中(Phase 1-3 改动) | 高(替换协作内核) |
| Excalidraw 兼容 | 已兼容 | Yjs + y-excalidraw 天然兼容 |
| 离线支持 | 需自建 | 开箱即用 |
| 冲突解决 | LWW(简单) | CRDT YATA(更精确) |
| 社区生态 | 无 | 大(编辑器集成多) |
| 学习曲线 | 低(自己的代码) | 中(Yjs CRDT 概念) |
| Flutter 支持 | 原生 Dart | 需 js 互操作或 port |

**建议**: 先完成 Phase 1-3 优化。如果优化后仍不满足需求，再评估 Yjs 集成。理由:
1. 当前 LWW 冲突策略对白板场景足够(Figma 也用 LWW)
2. Yjs 有成熟的 JS 实现但 Flutter 端需要 js bridge(增加复杂度)
3. 自研方案的痛点主要是工程实现问题(编码/批处理/连接)，非算法问题

## 实施建议

1. **Phase 1 优先**: 二进制传输 + 并行解密 = 最大收益，最小风险，2天内可完成
2. **Phase 2 验证后**: 批处理和取消全量同步需充分测试(尤其多人并发场景)
3. **Phase 3 逐步**: 连接稳定性改动涉及移动端 API，需在 Android + 鸿蒙两端验证
4. **每个 Phase 完成后**: 在 3+ 设备上进行多人协作压测，记录延迟和断连数据
5. **不建议**: 跳过 Phase 1 直接做 Yjs 集成——当前架构的 LWW 策略正确，问题在工程实现而非算法
